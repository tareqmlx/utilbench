# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Utilbench Web is a browser-based utility tools SPA. All tools run entirely client-side (no backend) using Web Workers and WASM for compute-heavy tasks. Built with React 19, React Router v7, Vite 6, Tailwind CSS 4, shadcn/ui, and TypeScript in strict mode. Package manager is Bun. Deployed on Cloudflare Workers Assets.

## Commands

- **Install deps:** `bun install`
- **Dev server:** `bun run dev`
- **Production build:** `bun run build` (chains: vite build → generate OG images → generate icons)
- **Vite build only:** `bun run build:only`
- **Preview build:** `bun run preview`
- **Lint & format check:** `bun run check` (runs Biome on `./src`)
- **Auto-fix lint/format:** `bunx --bun @biomejs/biome check --write ./src`
- **Tests (watch):** `bun run test`
- **Tests (single run):** `bun run test:run`
- **Tests (UI):** `bun run test:ui`

## Testing

Vitest with jsdom environment. Tests use `@testing-library/react` and `@testing-library/jest-dom` matchers. Setup file: `src/test/setup.ts`. Test files follow `src/**/*.test.{ts,tsx}` pattern, colocated in `__tests__/` directories alongside source.

The test setup (`src/test/setup.ts`) includes jsdom polyfills required for Radix UI: `ResizeObserver`, `PointerEvent`, pointer capture methods, and `scrollIntoView`. These are already in place — do not remove them or tests for Radix-based components will fail.

Auto-cleanup is **not enabled** — component tests must call `cleanup()` in `afterEach`. Tools using `useUrlState` (or any React Router hook) must wrap renders in `<MemoryRouter>` in tests.

## Architecture

### Path Alias

`@/*` maps to `./src/*`. Configured in both `tsconfig.app.json` and `vite.config.ts`. shadcn/ui components use this alias internally. Tool `Route.tsx` files use relative imports.

### UI Components (shadcn/ui)

- Components live in `src/components/ui/` — generated by shadcn CLI, customizable
- `cn()` utility in `src/lib/utils.ts` for merging Tailwind classes
- Theme uses CSS custom properties (oklch colors) in `src/index.css` with `:root` (light) and `.dark` (dark) scopes
- `components.json` at project root configures shadcn CLI

### Icons (Lucide React)

All icons use `lucide-react` — tree-shaken, no icon font. Tool icons are **statically mapped** in `src/lib/icons.ts` via `getIcon(name)`. When adding a new tool with a new icon, you must also add the Lucide import and mapping entry to `icons.ts`. Falls back to `Braces` for unknown names.

### Tool Plugin System

Tools are the core unit of the app. Each tool lives in `src/tools/<tool-slug>/` with two files:

- **`tool.ts`** — exports a **named** `tool` constant of type `ToolDefinition`:
  - `name`, `slug`, `description`, `category`, `tags`, `featured`, `icon`, `route`
  - `seoDescription?` — optional, used for meta tags when different from `description`
  - `icon` is a Lucide icon component name (string key into `src/lib/icons.ts`)
  - `category` is one of: `"media"` | `"data"` | `"text"`
  - `route` is a lazy import function: `() => import("./Route")`
- **`Route.tsx`** — default-exports a React component (lazy-loaded by the router)

Tools are auto-discovered at build time via `import.meta.glob` in `src/tools/registry.ts`. The glob expects a named export `tool` (not default export). To add a new tool, create its directory with those two files — no manual registration needed. Registry exports: `getAllTools()`, `getToolBySlug()`, `getFeaturedTools()`, `getToolsByCategory()`.

### Skeleton Loading

Each tool has a custom skeleton loader in `src/components/skeleton/skeletonRegistry.tsx` that displays while the lazy Route.tsx loads. Shared skeleton primitives (`SkeletonLine`, `SkeletonButton`, `SkeletonBlock`, `SkeletonTextArea`, `FeatureCardsSkeleton`) are in `src/components/skeleton/`. When adding a new tool, add a skeleton entry for it.

### Routing

`src/router.tsx` defines all routes with lazy loading. Tool pages are rendered at `/tools/:toolSlug` and wrapped in `ToolErrorBoundary`. Route handles carry `title` metadata for document titles.

### SEO

Uses `react-helmet-async` with `HelmetProvider` in `main.tsx`. Key modules in `src/seo/`:

- **`SEOHead`** — reusable component for title, description, OG, and Twitter Card meta tags
- **`JsonLd`** — injects JSON-LD structured data
- **`schemas.ts`** — builders for Organization, WebSite, SoftwareApplication, Breadcrumb, and WebPage schemas
- **`constants.ts`** — `SITE_URL` (defaults to `https://utilbench.devandstone.com`, override via `VITE_SITE_URL`), default OG image, default title/description

A custom Vite plugin in `vite.config.ts` auto-generates `sitemap.xml` during build from discovered tool slugs.

### Workers & WASM

- `src/workers/pool.ts` — Worker pool (2–4 threads based on `hardwareConcurrency`) with task queue, timeouts (30s default), and cancellation. Access via `workerPool` singleton from `src/workers`.
- `src/wasm/loader.ts` — Cached WASM module loader with streaming compilation fallback. Access via `instantiateWasm` from `src/wasm`.

### Tool Layout Primitives

Composable layout components in `src/components/tool-layout/` that every tool `Route.tsx` file uses. Barrel-exported from `index.ts`. Import via relative path from Route files (`../../components/tool-layout`).

- **`ToolShell`** — section wrapper used by every tool. Two variants: `variant="default"` (1320 px `wb-shell`) or `variant="wide"` (1900 px `max-w-300`, reserved for tools that legitimately need a larger work surface — currently `diff-checker` and `jwt-decoder`). Renders a `<section>`, never a `<main>` (the global `<main>` lives in `Layout.tsx`).
- **`PaneHeader`** — label (renders `<label>` when `htmlFor` is set, `<span>` otherwise) + icon + actions slot, used for input/output pane headers
- **`TwoPane`** — responsive two-column grid (`grid-cols-1 lg:grid-cols-2`) with configurable gap
- **`ErrorAlert`** — null-safe destructive alert; renders nothing when `error` is `null`

Every tool body wraps in `ToolShell`. Bespoke wrappers, decorative full-bleed backgrounds, or ad-hoc `max-w-*` overrides inside Route.tsx are forbidden.

### Custom Hooks

- **`useClipboard(resetDelay?)`** — async clipboard write/read with auto-resetting `copied` state for UI feedback
- **`useToolPreferences(slug, defaults)`** — localStorage-persisted tool settings with debounced writes, forward-compatible merge, and reset
- **`useUrlState(schema)`** — schema-based URL search param sync via React Router's `useSearchParams`. Supports `string`, `number`, `boolean` field types. Default values are omitted from URL for clean links. Values >1500 chars are not written (safety valve). Uses `{ replace: true }` to avoid history pollution. Currently integrated in cron-parser, base64-encoder, case-converter, and qr-generator. Components using this hook require a Router context (wrap with `MemoryRouter` in tests).
- **`useKeyboardShortcut(shortcuts)`** — declarative keyboard shortcut registration with meta/shift/enabled support

### Styling

Tailwind CSS 4 with shadcn/ui components and CSS custom properties for theming. Theme colors defined as CSS custom properties in `src/index.css` using oklch color space (with `:root` for light and `.dark` for dark — no in-app toggle currently). Use semantic color tokens (`bg-background`, `text-foreground`, `bg-card`, `text-muted-foreground`, `border-border`, etc.) instead of raw Tailwind colors. Fonts are self-hosted in `public/fonts/` (Inter variable).

### Global Search

`Cmd+K` opens a shadcn `CommandDialog` (cmdk-based) fuzzy search across all tools. Implemented in `src/components/SearchModal.tsx`. Uses Fuse.js with weighted keys: `name` (2), `description` (1), `tags` (1), `category` (0.5).

### Vendor Splitting

Vite config manually chunks: `vendor-react`, `vendor-router`, `vendor-fuse`, `vendor-lottie`, `vendor-svgo`, `vendor-radix` (all @radix-ui packages), `vendor-cmdk`.

### App Constants

`src/config.ts` exports `APP_NAME` and `APP_DESCRIPTION`.

### Tool Page Chrome

`src/pages/ToolPage.tsx` provides all shared chrome for every tool page. Individual `Route.tsx` files should only contain tool-specific functional UI. The following are centralized in `ToolPage.tsx` and must NOT be duplicated in any Route.tsx:

- **Breadcrumbs** — Home > Tools > Category > Tool navigation
- **Hero** — icon tile + H1 + description + stickers (`all-local`, category sticker)
- **`FeatureCards`** — three feature cards rendered from `tool.features`
- **Sibling tools** — three same-category tools at the bottom

When creating a new tool, the Route.tsx must start directly with the tool's interactive UI inside a `ToolShell` — no own `<h1>`, no breadcrumbs, no privacy / "client-side" badges, no decorative full-bleed backgrounds, no sibling-tool sections. Use the layout primitives from `src/components/tool-layout/` (`ToolShell`, `TwoPane`, `PaneHeader`, `ErrorAlert`) for consistent structure.

### Build Scripts

Scripts in `scripts/` run as part of the build pipeline:

- **`generate-og-image.ts`** — generates OG images using Satori + Resvg
- **`generate-icons.ts`** — converts `public/favicon.svg` to PNG icons (192px, 512px)
- **`generate-font-subset.ts`** — creates optimized font subsets
- **`prerender.ts`** — prerenders routes using Puppeteer

### Deployment (Cloudflare Workers Assets)

- SPA fallback handled by Wrangler config: `not_found_handling: "single-page-application"` (auto-generated `wrangler.jsonc`). Do **not** add a `public/_redirects` with `/* /index.html 200` — Cloudflare's parser flags it as an infinite loop because it strips `.html`/`/index` automatically.
- `public/_headers` — security headers (X-Frame-Options DENY, nosniff, strict referrer, permissions policy) and long-lived cache for assets/fonts. Cloudflare reads `_headers` the same as Netlify.
